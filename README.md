# TaskManager
Тестовое задание

## Запуск

Создайте файл `.env`, скопировав содержимое из `.env.example`:

Запустите проект:
```bash
docker compose up --build
```
 
Swagger UI: http://localhost:8000/docs

## API

| Метод | Путь | Описание |
|-------|------|----------|
| POST | /tasks/ | Создать задачу |
| GET | /tasks/?assignee={assignee}&status={status} | Список задач (фильтрация по `assignee`, `status`) |
| GET | /tasks/{task_id} | Получить задачу по ID |
| PATCH | /tasks/{task_id} | Обновить задачу |
| DELETE | /tasks/{task_id} | Удалить задачу |

## Статусы
`TODO` → `IN_PROGRESS` → `REVIEW` → `DONE`

Для статусов решил сделать Enum, тк они не должны меняться произвольно.
Pydantic отклоняет невалидный статус на входе, PostgreSQL не даёт записать его в бд. 
`StrEnum` вместо `Enum` убирает необходимость обращаться к `.value` при сравнениях.


Переход назад из `DONE` невозможен - сервис возвращает `400` с ошибкой.

## Модель задачи

| Поле | Тип | Описание |
|------|-----|----------|
| id | int | Первичный ключ |
| title | string | Название задачи |
| description | string | Описание |
| assignee | string | Исполнитель |
| status | enum | Текущий статус |
| created_at | datetime | Дата создания |
| updated_at | datetime | Дата последнего изменения 


`created_at` и `updated_at` добавлены для аудита и отслеживания задачи.

## Дополнительные библиотеки

**PostgreSQL** - для хранения задач в базе данных с чёткой структурой.  

**SQLAlchemy** - ORM для удобной работы с SQL через python.  

**Redis** - для кэширования списка задач без фильтров.

## Кэширование

Список задач (`GET /tasks/` без фильтров) кэшируется в Redis с TTL 60 секунд.

**Обоснование TTL:**
- Список задач - относительно стабильные данные, массовые изменения происходят редко, чтение списка - часто
- 60 секунд - если брать меньше, то в кеше нет смысла, а если брать больше, то можно получить устаревший список
- При любой мутации (create / update / delete) кэш инвалидируется

## Тесты

```bash
pytest -v
```
Покрытие:
- Создание задачи
- Валидация входных данных
- Смена статуса
- Запрет перехода назад из `DONE`
- Получение несуществующей задачи

## Архитектура

Проект разделён на слои (слоистая архитектура):

- **Routers** - принимает HTTP-запросы, валидирует входные данные через Pydantic-схемы
- **Service** - бизнес логика
- **Repository** - работа с БД через SQLAlchemy, без бизнес-логики
- **Schema** - Pydantic модели для валидации входных и выходных данных
- **Database** - SQLAlchemy модели и конфигурация подключения к БД

Такое разделение позволяет менять реализацию каждого слоя независимо
